# 架构决策记录

每次重大决策追加一条，格式：背景 → 备选方案 → 选择理由 → 后果。修 bug、加小功能不记。

---

### ADR-001：事件文本用 .md 文件而非代码硬编码

**日期：** 2025 (v0.x 初期)

**背景：** 需要让创作者（非技术背景）能独立创作副本内容，不依赖改代码。

**方案 A：** 代码内置 — 每次改剧情要改代码、重新打包。
**方案 B：** .md 文件 + build.js 解析 — 内容与代码分离，构建时生成 JSON 嵌入。

**决策：** 选 B。创作者写 .md 即可，不需要碰代码或重新编译。

**后果：** 新增副本只需写 .md 文件，运行 node build.js 即可；但也增加了构建步骤的复杂度。

### ADR-002：前端无框架（纯 HTML/CSS/JS）

**日期：** 2025 (v0.x 初期)

**背景：** 项目从 Tauri (Rust 前端) 迁移到 Electron，需要重新选择前端方案。

**方案 A：** 引入 React/Vue — 社区成熟，组件化开发；但增加打包体积和学习成本。
**方案 B：** 纯 HTML/CSS/JS — 零额外依赖，直接满足需求。

**决策：** 选 B。项目 UI 复杂度不高（面板 + 列表 + 弹窗），框架带来的价值有限，零依赖意味着零兼容问题和零更新负担。

**后果：** 前端代码集中在 3 个文件（main.js / style.css / index.html），维护简便。

### ADR-003：Hub + Scenario 双层架构

**日期：** 2025 (v0.x 初期)

**背景：** 需要区分"管理"和"游玩"两个状态——大厅里选副本、管理历程，副本内挂机升级。

**方案 A：** 单层（全局游戏循环）— 任何时候都在跑游戏逻辑，状态管理复杂。
**方案 B：** 双层（大厅/副本分离）— 大厅纯管理界面，进入副本后才启动游戏循环。

**决策：** 选 B。副本进出时状态切换清晰（出副本重置等级、EXP 累入大厅），大厅内不涨 EXP、不触发事件、不检测成就，逻辑隔离干净。

**后果：** 游戏引擎需要处理 isInHub 状态判断；所有 EXP/事件逻辑集中在"非大厅"分支。

### ADR-004：存档用本地 JSON 文件而非数据库

**日期：** 2025 (v0.x 初期)

**背景：** 单机本地应用，存档数据量小（单个玩家，纯文本），需要持久化。

**方案 A：** SQLite — 强一致性、事务支持；但引入数据库管理和跨平台编译复杂度。
**方案 B：** JSON 文件 — 人类可读，JSON.parse/stringify 即可读写，无需额外依赖。

**决策：** 选 B。数据量不足以支撑数据库开销，JSON 可读、可手动编辑、可 git diff。后续加了原子写入（.tmp + rename）解决写半截崩溃的问题。

**后果：** 存档可读性强，但也容易被用户手动修改（v3.4.3 加了固定 tick 计数器防止时间篡改）。

### ADR-005：从 node-global-key-listener 替换为 uiohook-napi

**日期：** 2026-06 (v3.3.0)

**背景：** node-global-key-listener 依赖 WinKeyServer.exe 实现全局键盘监听，Windows Defender 将其误报为木马，影响用户信任。

**方案 A：** 写白名单说明文档 — 治标不治本，用户仍然不敢用。
**方案 B：** uiohook-napi — 原生 Node.js 模块，直接监听系统输入，不依赖额外 exe。

**决策：** 选 B。彻底消除误报源，且支持鼠标监听（左/右/中键可同时计入连击和 EXP）。

**后果：** uiohook-napi 需要 node-gyp 编译，CI 三平台需额外依赖（X11 头文件、Python setuptools、VS Build Tools）；CI runner 需要固定版本号避免镜像变动。

### ADR-006：Windows exe 图标分环境注入策略

**日期：** 2026-06 (v3.1.4)

**背景：** Windows 构建的 exe 图标始终显示 Electron 默认图标，rcedit/electron-builder 注入均返回 OK 但图标不变。

**根因：** signAndEditExecutable: false 同时禁用了图标注入（electron-builder 设计缺陷 #6934）；旧 BMP-in-ICO 格式兼容性差；构建后注入时机不对。

**方案 A：** 不分环境，全用 post-inject 脚本 — 本地有效，但 GitHub Actions 无管理员权限时 rcedit 不可靠。
**方案 B：** 分环境策略 — GitHub Actions 设 signAndEditExecutable=true 让 builder 自己注入；本地用 post-inject + MD5 验证 + 3 次重试。

**决策：** 选 B。分环境各取所长：CI 有权限走 builder 内置注入，本地无权限走脚本兜底。

**后果：** 维护两套注入路径；icons/ 目录需要 7 尺寸 .ico 合并为 icon_png.ico；踩坑经验已沉淀为 CI/CD 配置规则。
