# ScenarioWriter — Idel-DreamMaker 分支人生副本生成

## 这是什么

为挂机游戏 Idel-DreamMaker 创作副本（剧本）。每个副本 = 一次完整的人生经历，500 个等级，每升一级弹出故事。关键节点还可以做选择，影响后续剧情走向。

## 数据规格

| 项目 | 值 |
|------|-----|
| 普通 story | ~475/周目 |
| choice 事件 | 20-25/周目（关键节点让你选） |
| choice target | 40-50/周目（选完后的分支剧情） |
| filler 日常 | 1365/周目 |
| 事件类型 | story（剧情）/ filler（日常） |
| 文本长度 | 30-80 字 |
| 叙事人称 | 第二人称"你" |
| 称号 | 30 个/副本 |
| 成就 | 50 个/副本 |
| 节日事件 | 约 52 条/副本 |
| 多周目重生 | 最多 4 周目（主线+番外+另一视角+隐藏结局） |
| 挂机机制 | standard / cultivation / cyber / tide / polar 五种 |

## 游戏机制简介

- 等级曲线：Lv.1-100 用平方根公式，Lv.100+ 线性加速
- 升到 Lv.500 约需 117 天（每天挂 8 小时）
- Lv.500 触发结局弹窗
- 重生后经验加成 +10%（封顶 +50%），filler 上限 +5（封顶 +25）

## 工作流程

### Stage 1：确定副本设定

1. 问用户想写什么主题的副本（废土、修仙、赛博、日常……什么都可以）
2. 理解世界观设定
3. 推荐挂机机制（适合的 mechanic）
4. 确定以下元数据：

```
id:              小写英文+数字+下划线
name:            英文名
name_cn:         中文名
description:     一句话世界观描述
player_title:    Lv.1 默认称号
mechanic:        挂机机制
max_rebirth:     最大重生次数（默认 3）
completion_title:通关称号
```

输出预览并等用户确认。

### Stage 1.5：定大纲 + choice 规划

先写每个周目的故事大纲（50-100 字），然后同步规划 choice 节点。

**默认节奏：**

| 阶段 | 等级 | 风格 |
|------|------|------|
| 开端 | 1-60 | 初始世界、初遇挑战 |
| 成长 | 61-140 | 结交盟友、确立目标 |
| 鼎盛 | 141-300 | 建功立业、应对大敌 |
| 沉淀 | 301-450 | 培养后辈、准备谢幕 |
| 终章 | 451-500 | 完成最后之事、坦然离去 |

**每周目 choice 数量规划（可调整）：**

| 周目 | choice 总数 | 重大抉择 | 快速选择 |
|:----:|:----------:|:--------:|:--------:|
| R0（主线） | 25 | 6 | 19 |
| R1（番外） | 22 | 6 | 16 |
| R2（另一视角） | 20 | 5 | 15 |
| R3（隐藏结局） | 18 | 5 | 13 |

**两种 choice：**
- **重大抉择**（每周目 5-6 个）：选了之后影响后续 15 级的故事走向。放关键转折点。
- **快速选择**（每周目 13-19 个）：只影响下一条事件，小插曲。放日常节点。

**输出格式：**
```
周目0 choice 规划：
  Lv.20  初遇流浪者 → 收留 / 赶走（快速）
  Lv.50  岔路口 → 走大道 / 穿密林（重大，影响 Lv.51-65）
  Lv.80  匪徒索要物资 → 给 / 打（重大，影响 Lv.81-95）
```

等用户确认 choice 规划后进入下一步。

### Stage 1.6：设定锚点 + 设计 filler 模板

设计副本的世界观锚点（角色、地点、物件、专属名词），然后设计 filler 日常事件的句式模板。

停顿，等用户确认锚点和模板。

### Stage 2：分段生成

一周目拆成 5 段生成，每段 260-420 条事件。每完成一段输出进度。

**灯停规则：** 每个周目的 5 段全部生成后，跑一次 `node build.js` 验证。通过则继续下一周目，失败则停住贴错误信息。

**ID 命名规则：**
- 普通 story：`{id}_e{四位编号}`（如 `wasteland_e0050`）
- choice 事件：与普通 story 同级编号，加 Choice 列
- target A 事件：`{id}_e{编号}a`（如 `wasteland_e0050a`）
- target B 事件：`{id}_e{编号}b`（如 `wasteland_e0050b`）

**事件表格式：**

```markdown
## Events

| ID | Type | MinLevel | MinHours | Weight | Once | MinRebirth | Action | Choice1 | Choice1Target | Choice2 | Choice2Target | Text |
|---|---|---|---|---|---|---|---|---|---|---|---|---|
| {id}_e0001 | story | 1 | 0 | 1 | 是 | 0 | 觉醒 | | | | | 你睁开眼睛... |
| {id}_e0050 | story | 50 | 0 | 1 | 是 | 0 | 抉择 | 走大道 | e0050a | 穿密林 | e0050b | 你站在岔路口... |
| {id}_e0050a | story | 51 | 0 | 1 | 是 | 0 | 行路 | | | | | 你走上大道... |
```

### Stage 3：写入文件

文件放到项目目录的 `scenarios/` 下。

### Stage 4：验证

```bash
node build.js
```

如验证失败，根据错误信息修复后重新验证。

## 注意事项（避免翻车）

- 文件第一行必须是 `---`（不能有 BOM）
- Events 表内部**不能有空行**
- 每行 pipe table 必须以 `|` 开头和结尾
- choice target 事件 ID 必须在 Events 表中存在
- 事件文本不少于 10 字
- 所有文本用第二人称"你"写
- 不同周目的 filler 不能重复
